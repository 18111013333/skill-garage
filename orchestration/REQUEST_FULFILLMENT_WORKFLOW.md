# REQUEST_FULFILLMENT_WORKFLOW.md - 服务请求履行流程

**版本: V27.0**

## 目的
定义服务请求履行流程，确保服务请求可标准处理，不再散在聊天和工单里，实现"请求有标准、处理有流程、结果可追踪"。

## 适用范围
所有服务请求场景，包括：
- 服务开通请求
- 资源申请请求
- 权限申请请求
- 配置变更请求
- 支持请求

---

## 一、请求类型

| 类型 | 代码 | 说明 | 处理时限 |
|------|------|------|----------|
| 开通请求 | PROVISION | 服务开通 | 24 小时 |
| 资源请求 | RESOURCE | 资源申请 | 48 小时 |
| 权限请求 | PERMISSION | 权限申请 | 24 小时 |
| 配置请求 | CONFIG | 配置变更 | 48 小时 |
| 支持请求 | SUPPORT | 技术支持 | 按优先级 |

---

## 二、请求流程

```
请求提交
    ↓
┌─────────────────────────────────────┐
│ 1. 请求受理                          │
│    - 接收请求                        │
│    - 验证请求格式                    │
│    - 分配请求ID                      │
└─────────────────────────────────────┘
    ↓
┌─────────────────────────────────────┐
│ 2. 请求分类                          │
│    - 识别请求类型                    │
│    - 评估优先级                      │
│    - 确定处理路径                    │
└─────────────────────────────────────┘
    ↓
┌─────────────────────────────────────┐
│ 3. 自动处理（如适用）                │
│    - 检查是否可自动处理              │
│    - 执行自动处理                    │
│    - 记录处理结果                    │
└─────────────────────────────────────┘
    ↓
┌─────────────────────────────────────┐
│ 4. 审批节点（如需要）                │
│    - 路由到审批人                    │
│    - 等待审批结果                    │
│    - 处理审批结果                    │
└─────────────────────────────────────┘
    ↓
┌─────────────────────────────────────┐
│ 5. 人工介入（如需要）                │
│    - 分配处理人                      │
│    - 执行处理                        │
│    - 记录处理过程                    │
└─────────────────────────────────────┘
    ↓
┌─────────────────────────────────────┐
│ 6. 完成确认                          │
│    - 验证处理结果                    │
│    - 通知请求人                      │
│    - 关闭请求                        │
└─────────────────────────────────────┘
    ↓
┌─────────────────────────────────────┐
│ 7. 超时升级（如超时）                │
│    - 检测超时                        │
│    - 自动升级                        │
│    - 通知升级目标                    │
└─────────────────────────────────────┘
```

## 三、请求受理

### 3.1 受理渠道

| 渠道 | 说明 | 响应时间 |
|------|------|----------|
| API | 通过API提交 | 实时 |
| 控制台 | 通过控制台提交 | 实时 |
| 邮件 | 通过邮件提交 | 1 小时 |
| IM | 通过IM提交 | 30 分钟 |

### 3.2 受理检查

| 检查项 | 说明 | 失败处理 |
|--------|------|----------|
| 格式验证 | 验证请求格式 | 返回格式错误 |
| 权限验证 | 验证请求权限 | 返回权限不足 |
| 配额验证 | 验证配额限制 | 返回配额不足 |
| 重复检测 | 检测重复请求 | 返回重复提示 |

### 3.3 受理确认

```
【请求受理确认】
请求ID: {request_id}
请求类型: {request_type}
提交时间: {submit_time}
预计处理时间: {estimated_time}
当前状态: {status}
```

## 四、请求分类

### 4.1 分类规则

| 请求类型 | 分类条件 | 处理路径 |
|----------|----------|----------|
| 开通请求 | 包含"开通"、"创建" | 自动处理 |
| 资源请求 | 包含"资源"、"配额" | 审批后处理 |
| 权限请求 | 包含"权限"、"授权" | 审批后处理 |
| 配置请求 | 包含"配置"、"修改" | 审批后处理 |
| 支持请求 | 包含"问题"、"故障" | 人工处理 |

### 4.2 优先级评估

| 优先级 | 条件 | 处理时限 |
|--------|------|----------|
| P0 紧急 | 业务阻塞 | 1 小时 |
| P1 高 | 严重影响 | 4 小时 |
| P2 中 | 一般影响 | 24 小时 |
| P3 低 | 轻微影响 | 72 小时 |

### 4.3 处理路径确定

| 路径 | 条件 | 说明 |
|------|------|------|
| 自动处理 | 标准化请求 + 无需审批 | 自动执行 |
| 审批后处理 | 需要审批 | 审批后执行 |
| 人工处理 | 复杂请求 | 人工处理 |

---

## 五、自动处理

### 5.1 可自动处理请求

| 请求类型 | 自动处理条件 | 处理动作 |
|----------|--------------|----------|
| 标准服务开通 | 标准服务 + 配额内 | 自动开通 |
| 标准资源申请 | 标准资源 + 配额内 | 自动分配 |
| 标准权限申请 | 标准权限 + 已授权 | 自动授权 |
| 标准配置变更 | 标准配置 + 低风险 | 自动变更 |

### 5.2 自动处理流程

```
自动处理触发
    ↓
┌─────────────────────────────────────┐
│ 1. 前置验证                          │
│    - 验证处理条件                    │
│    - 验证资源可用性                  │
└─────────────────────────────────────┘
    ↓
┌─────────────────────────────────────┐
│ 2. 执行处理                          │
│    - 执行处理动作                    │
│    - 记录处理日志                    │
└─────────────────────────────────────┘
    ↓
┌─────────────────────────────────────┐
│ 3. 后置验证                          │
│    - 验证处理结果                    │
│    - 确认服务正常                    │
└─────────────────────────────────────┘
    ↓
┌─────────────────────────────────────┐
│ 4. 通知确认                          │
│    - 通知请求人                      │
│    - 更新请求状态                    │
└─────────────────────────────────────┘
```

### 5.3 自动处理限制

| 限制项 | 说明 |
|--------|------|
| 配额限制 | 不超过配额限制 |
| 频率限制 | 不超过频率限制 |
| 风险限制 | 不超过风险等级 |

## 六、审批节点

### 6.1 审批触发条件

| 条件 | 审批级别 | 说明 |
|------|----------|------|
| 超配额 | 服务Owner | 超过配额限制 |
| 高风险 | 技术负责人 | 高风险操作 |
| 跨租户 | 技术总监 | 跨租户操作 |
| 核心服务 | 技术总监 | 涉及核心服务 |

### 6.2 审批流程

```
审批触发
    ↓
┌─────────────────────────────────────┐
│ 1. 审批请求                          │
│    - 生成审批请求                    │
│    - 附带请求详情                    │
└─────────────────────────────────────┘
    ↓
┌─────────────────────────────────────┐
│ 2. 审批路由                          │
│    - 确定审批人                      │
│    - 发送审批通知                    │
└─────────────────────────────────────┘
    ↓
┌─────────────────────────────────────┐
│ 3. 等待审批                          │
│    - 等待审批结果                    │
│    - 超时自动升级                    │
└─────────────────────────────────────┘
    ↓
┌─────────────────────────────────────┐
│ 4. 处理结果                          │
│    - 通过：继续处理                  │
│    - 拒绝：通知请求人                │
│    - 驳回：返回修改                  │
└─────────────────────────────────────┘
```

### 6.3 审批时限

| 审批级别 | 时限 | 超时处理 |
|----------|------|----------|
| 服务Owner | 4 小时 | 自动升级 |
| 技术负责人 | 8 小时 | 自动升级 |
| 技术总监 | 24 小时 | 自动拒绝 |

## 七、人工介入

### 7.1 人工介入条件

| 条件 | 说明 |
|------|------|
| 复杂请求 | 无法自动处理的复杂请求 |
| 异常情况 | 处理过程中出现异常 |
| 特殊需求 | 有特殊处理需求 |
| 人工请求 | 请求人工处理 |

### 7.2 人工处理流程

```
人工介入触发
    ↓
┌─────────────────────────────────────┐
│ 1. 任务分配                          │
│    - 分配处理人                      │
│    - 通知处理人                      │
└─────────────────────────────────────┘
    ↓
┌─────────────────────────────────────┐
│ 2. 问题分析                          │
│    - 分析请求内容                    │
│    - 制定处理方案                    │
└─────────────────────────────────────┘
    ↓
┌─────────────────────────────────────┐
│ 3. 执行处理                          │
│    - 执行处理动作                    │
│    - 记录处理过程                    │
└─────────────────────────────────────┘
    ↓
┌─────────────────────────────────────┐
│ 4. 结果确认                          │
│    - 确认处理结果                    │
│    - 通知请求人                      │
└─────────────────────────────────────┘
```

### 7.3 处理人分配

| 请求类型 | 分配规则 | 说明 |
|----------|----------|------|
| 开通请求 | 服务团队 | 按服务分配 |
| 资源请求 | 运维团队 | 按资源类型分配 |
| 权限请求 | 安全团队 | 按权限类型分配 |
| 配置请求 | 服务团队 | 按服务分配 |
| 支持请求 | 支持团队 | 按优先级分配 |

## 九、超时升级

### 9.1 超时检测

| 检测点 | 说明 | 检测频率 |
|--------|------|----------|
| 受理超时 | 受理阶段超时 | 每分钟 |
| 处理超时 | 处理阶段超时 | 每5分钟 |
| 审批超时 | 审批阶段超时 | 每10分钟 |
| 确认超时 | 确认阶段超时 | 每小时 |

### 9.2 升级路径

| 阶段 | 第一级 | 第二级 | 第三级 |
|------|--------|--------|--------|
| 受理 | 值班人员 | 服务Owner | 技术负责人 |
| 处理 | 处理人 | 服务Owner | 技术负责人 |
| 审批 | 审批人 | 上级审批人 | 技术总监 |
| 确认 | 处理人 | 服务Owner | 技术负责人 |

### 9.3 升级通知

```
【超时升级通知】
请求ID: {request_id}
当前阶段: {current_stage}
超时时间: {timeout_time}
升级目标: {escalation_target}
升级原因: {escalation_reason}
```

## 十、请求追踪

### 10.1 状态定义

| 状态 | 代码 | 说明 |
|------|------|------|
| 已提交 | SUBMITTED | 请求已提交 |
| 已受理 | ACCEPTED | 请求已受理 |
| 处理中 | PROCESSING | 请求处理中 |
| 待审批 | PENDING_APPROVAL | 等待审批 |
| 待确认 | PENDING_CONFIRM | 等待确认 |
| 已完成 | COMPLETED | 请求已完成 |
| 已拒绝 | REJECTED | 请求已拒绝 |
| 已取消 | CANCELLED | 请求已取消 |

### 10.2 追踪接口

```
# 查询请求状态
GET /requests/{request_id}/status

# 查询请求详情
GET /requests/{request_id}

# 查询请求历史
GET /requests/{request_id}/history
```
