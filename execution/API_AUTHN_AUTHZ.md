# API_AUTHN_AUTHZ.md - API 认证与授权规则

## 目的
定义 API 认证与授权机制，确保接口安全可控。

## 适用范围
所有对外公开的 API 接口。

---

## 一、认证模式

### 1.1 API Key 认证
**适用场景**: 服务端调用、自动化任务

```
请求头: Authorization: Bearer <api_key>
```

| 属性 | 说明 |
|------|------|
| 格式 | `sk_live_xxx` 或 `sk_test_xxx` |
| 有效期 | 永久（可手动撤销） |
| 权限范围 | 绑定租户和应用 |
| 存储 | 加密存储，不记录明文 |

### 1.2 OAuth 2.0 认证
**适用场景**: 用户授权、第三方应用

| 授权模式 | 用途 |
|----------|------|
| Authorization Code | Web 应用 |
| Client Credentials | 服务间调用 |
| Device Code | 设备端应用 |
| Refresh Token | 长期访问 |

**OAuth 流程**:
```
用户授权 → 获取 code → 交换 token → 调用 API
              ↓            ↓
         redirect_uri   access_token
```

### 1.3 Service Account 认证
**适用场景**: 平台内部服务、自动化任务

| 属性 | 说明 |
|------|------|
| 身份类型 | 非用户身份 |
| 权限范围 | 绑定服务角色 |
| 审计追踪 | 记录服务来源 |
| 密钥轮换 | 支持 |

## 二、授权范围

### 2.1 Tenant Scope（租户范围）
| Scope | 说明 | 示例 |
|-------|------|------|
| `tenant:read` | 读取租户信息 | 查看配置 |
| `tenant:write` | 修改租户设置 | 更新配置 |
| `tenant:admin` | 租户管理权限 | 管理成员 |

### 2.2 Role Scope（角色范围）
| Scope | 说明 | 权限 |
|-------|------|------|
| `role:viewer` | 只读角色 | GET 请求 |
| `role:editor` | 编辑角色 | GET/POST/PUT |
| `role:admin` | 管理角色 | 所有操作 |
| `role:platform` | 平台角色 | 跨租户操作 |

### 2.3 API Scope（接口范围）
| Scope | 说明 |
|-------|------|
| `api:chat` | 对话接口权限 |
| `api:memory` | 记忆接口权限 |
| `api:skills` | 技能接口权限 |
| `api:connectors` | 连接器接口权限 |
| `api:marketplace` | 市场接口权限 |
| `api:analytics` | 分析接口权限 |

## 三、委托访问

### 3.1 代用户访问
**场景**: 应用代表用户执行操作

```
应用 → 用户授权 → 获取 delegated token → 以用户身份调用
```

| 要求 | 说明 |
|------|------|
| 用户同意 | 必须显式授权 |
| 权限最小化 | 只请求必要 scope |
| 审计记录 | 记录代理关系 |
| 可撤销 | 用户可随时撤销 |

### 3.2 代租户访问
**场景**: 平台服务代表租户执行操作

```
平台服务 → Service Account → 以租户身份调用
```

| 要求 | 说明 |
|------|------|
| 租户授权 | 租户必须启用该服务 |
| 权限绑定 | 服务权限预定义 |
| 审计记录 | 记录服务来源 |
| 可禁用 | 租户可禁用服务 |

## 四、Token 管理

### 4.1 Token 属性
| 属性 | 说明 |
|------|------|
| access_token | 访问令牌，有效期短 |
| refresh_token | 刷新令牌，有效期长 |
| expires_in | 过期时间（秒） |
| token_type | 令牌类型（Bearer） |
| scope | 授权范围 |

### 4.2 Token 有效期
| Token 类型 | 有效期 | 说明 |
|------------|--------|------|
| Access Token | 1 小时 | 短期访问 |
| Refresh Token | 30 天 | 长期刷新 |
| API Key | 永久 | 手动管理 |

### 4.3 Token 失效
| 失效原因 | 处理 |
|----------|------|
| 过期 | 使用 refresh_token 刷新 |
| 撤销 | 需重新授权 |
| 权限变更 | 需重新授权 |
| 安全事件 | 强制失效 |

---

## 五、最小权限原则

### 5.1 权限请求
```
只请求必要的 scope，不请求全部权限
```

### 5.2 权限验证
```
每次请求验证: 身份 + scope + 资源权限
```

### 5.3 权限提升
```
需要更高权限时，要求重新授权
```

## 六、审计要求

### 6.1 审计日志
| 字段 | 说明 |
|------|------|
| timestamp | 请求时间 |
| caller_id | 调用者身份 |
| caller_type | 用户/服务/应用 |
| tenant_id | 租户 ID |
| api_id | 接口 ID |
| scope | 授权范围 |
| result | 授权结果 |

### 6.2 审计事件
| 事件 | 记录 |
|------|------|
| Token 签发 | ✅ |
| Token 刷新 | ✅ |
| Token 撤销 | ✅ |
| 授权失败 | ✅ |
| 权限提升 | ✅ |

---

## 七、安全最佳实践

### 7.1 密钥保护
- 不在代码中硬编码
- 使用环境变量或密钥管理服务
- 定期轮换密钥

### 7.2 传输安全
- 全程 HTTPS
- 禁止明文传输密钥
- 使用安全头部

### 7.3 错误处理
- 不暴露敏感信息
- 统一错误格式
- 记录异常请求
