# ESCALATION_MATRIX.md - 事故升级矩阵

**版本: V27.0**

## 目的
定义事故升级矩阵，确保事故不会卡在单点，实现"升级有规则、超时有处理、责任有归属"。

## 适用范围
所有事故升级场景，包括：
- 工具异常升级
- 租户问题升级
- 权限问题升级
- 事件处理异常升级
- 外部编排异常升级

---

## 一、升级原则

| 原则 | 说明 |
|------|------|
| 及时升级 | 超时即升级，不等待 |
| 逐级升级 | 按层级逐级升级 |
| 信息完整 | 升级时传递完整信息 |
| 记录可查 | 升级过程有记录 |
| 回归闭环 | 问题解决后回归闭环 |

## 四、升级路径

### 4.1 技术升级路径

```
值班人员 → 技术负责人 → 技术经理 → 技术总监 → CTO → CEO
```

### 4.2 安全升级路径

```
安全工程师 → 安全负责人 → 安全总监 → CTO → CEO
```

### 4.3 运维升级路径

```
运维工程师 → 运维负责人 → 运维经理 → 技术总监 → CTO
```

### 4.4 客户升级路径

```
客服 → 客户成功经理 → 客户成功总监 → VP → CEO
```

## 五、升级流程

```
升级触发
    ↓
┌─────────────────────────────────────┐
│ 1. 升级判断                          │
│    - 判断升级条件是否满足            │
│    - 确定升级目标                    │
└─────────────────────────────────────┘
    ↓
┌─────────────────────────────────────┐
│ 2. 信息准备                          │
│    - 收集事故信息                    │
│    - 准备升级说明                    │
│    - 附带处理历史                    │
└─────────────────────────────────────┘
    ↓
┌─────────────────────────────────────┐
│ 3. 升级通知                          │
│    - 通知升级目标                    │
│    - 通知原处理人                    │
│    - 通知相关方                      │
└─────────────────────────────────────┘
    ↓
┌─────────────────────────────────────┐
│ 4. 接手确认                          │
│    - 升级目标确认接手                │
│    - 更新处理人                      │
│    - 记录升级日志                    │
└─────────────────────────────────────┘
    ↓
┌─────────────────────────────────────┐
│ 5. 继续处理                          │
│    - 新处理人继续处理                │
│    - 原处理人协助                    │
└─────────────────────────────────────┘
```

## 六、升级超时处理

### 6.1 超时判定

| 超时类型 | 说明 | 处理方式 |
|----------|------|----------|
| 响应超时 | 升级目标未响应 | 继续升级 + 告警 |
| 接手超时 | 升级目标未接手 | 继续升级 + 告警 |
| 处理超时 | 处理时限超时 | 继续升级 + 告警 |

### 6.2 超时升级规则

| 当前层级 | 超时后升级到 | 超时时间 |
|----------|--------------|----------|
| 值班人员 | 技术负责人 | 15 分钟 |
| 技术负责人 | 技术经理 | 30 分钟 |
| 技术经理 | 技术总监 | 1 小时 |
| 技术总监 | CTO | 2 小时 |
| CTO | CEO | 4 小时 |

### 6.3 超时告警

| 告警类型 | 通知方式 | 通知范围 |
|----------|----------|----------|
| 响应超时 | 电话 + 短信 | 升级目标 + 上级 |
| 接手超时 | 电话 + 短信 | 升级目标 + 上级 |
| 处理超时 | 电话 + 短信 | 升级目标 + 上级 + 管理层 |

## 七、升级信息传递

### 7.1 必传信息

| 信息项 | 说明 |
|--------|------|
| 事故ID | 事故唯一标识 |
| 事故等级 | 严重等级 |
| 事故描述 | 事故现象描述 |
| 影响范围 | 影响的用户/系统 |
| 处理历史 | 已采取的处理措施 |
| 当前状态 | 当前处理状态 |
| 升级原因 | 升级的原因 |
| 期望支持 | 期望的支持 |

### 7.2 信息格式

```
【事故升级通知】
事故ID: {incident_id}
等级: {severity}
描述: {description}
影响: {impact}
处理历史: {history}
当前状态: {status}
升级原因: {reason}
期望支持: {expectation}
升级时间: {escalation_time}
```

## 八、升级统计

### 8.1 统计指标

| 指标 | 说明 | 目标 |
|------|------|------|
| 升级率 | 需升级的事故比例 | < 20% |
| 升级及时率 | 及时升级的比例 | > 95% |
| 接手及时率 | 及时接手的比例 | > 95% |
| 升级解决率 | 升级后解决的比例 | > 90% |

### 8.2 升级报告

| 报告类型 | 频率 | 内容 |
|----------|------|------|
| 升级统计 | 每周 | 升级情况统计 |
| 升级分析 | 每月 | 升级原因分析 |
| 升级改进 | 每季度 | 升级流程改进 |

## 九、异常处理

| 异常类型 | 处理方式 |
|----------|----------|
| 升级目标不可达 | 继续升级到下一级 |
| 升级目标拒绝 | 强制升级 + 记录 |
| 升级信息不完整 | 补充信息后升级 |
| 升级循环 | 终止 + 最高层裁决 |

| 标准 | 验证方式 |

| 升级及时 | 超时即升级 |
| 信息完整 | 升级信息完整 |
| 接手确认 | 接手有确认 |
