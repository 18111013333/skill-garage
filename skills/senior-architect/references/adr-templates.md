# 架构决策记录 (ADR) 模板库

## 什么是ADR?

架构决策记录(Architecture Decision Record)是一种记录重要架构决策的轻量级文档格式。它帮助团队:
- 追溯决策背景
- 理解权衡取舍
- 避免重复讨论
- 新成员快速了解系统

---

## ADR模板

### 模板1: 简化版 (Y-Statement)

适用于快速记录简单决策。

```markdown
# ADR-NNN: [决策标题]

在 [上下文/背景] 的情况下，面对 [问题/约束]，我们决定 [决策内容]，
以获得 [收益]，接受 [代价/风险]。

## 状态
[提议中 | 已接受 | 已废弃 | 已替代]

## 日期
YYYY-MM-DD

## 决策者
[参与决策的人员]
```

**示例**:
```markdown
# ADR-001: 使用PostgreSQL作为主数据库

在构建电商平台的背景下，面对需要ACID事务和复杂查询的需求，
我们决定使用PostgreSQL作为主数据库，以获得数据一致性和成熟的SQL支持，
接受需要单独处理水平扩展的代价。

## 状态
已接受

## 日期
2024-01-15

## 决策者
张三, 李四, 王五
```

## 云原生ADR模板

### ADR: 容器编排平台选择

```markdown
# ADR-XXX: 容器编排平台选择

## 状态
已接受

## 背景
我们需要选择容器编排平台来部署微服务架构，支持:
- 自动扩缩容
- 服务发现
- 负载均衡
- 滚动更新
- 自愈能力

## 决策
选择 **Kubernetes** 作为容器编排平台。

## 选项分析

### Kubernetes
- 优点:
  - 行业标准，生态丰富
  - 强大的调度能力
  - 多云支持
  - 社区活跃
- 缺点:
  - 学习曲线陡峭
  - 运维复杂度高
  - 资源开销大

### Docker Swarm
- 优点:
  - 简单易用
  - 与Docker原生集成
  - 轻量级
- 缺点:
  - 功能有限
  - 社区活跃度下降
  - 企业支持弱

### Nomad
- 优点:
  - 简单灵活
  - 支持非容器工作负载
  - HashiCorp生态
- 缺点:
  - 生态较小
  - 社区规模小

## 决策理由
1. 团队已有K8s经验
2. 云厂商原生支持(AKS/EKS/GKE)
3. 丰富的工具生态(Helm, Istio等)
4. 长期技术投资价值

## 后果
- 需要培训团队K8s知识
- 初期部署时间增加
- 长期运维成本可控
```

### ADR: 服务网格选择

```markdown
# ADR-XXX: 服务网格技术选型

## 状态
已接受

## 背景
微服务数量增长到50+，需要统一管理:
- 服务间通信
- 流量控制
- 可观测性
- 安全通信

## 决策
选择 **Istio** 作为服务网格解决方案。

## 选项分析

| 维度 | Istio | Linkerd | Consul Connect |
|------|-------|---------|----------------|
| 功能完整性 | ★★★★★ | ★★★ | ★★★★ |
| 性能开销 | ★★★ | ★★★★★ | ★★★★ |
| 易用性 | ★★★ | ★★★★★ | ★★★★ |
| 社区规模 | ★★★★★ | ★★★ | ★★★ |
| K8s集成 | ★★★★★ | ★★★★★ | ★★★★ |

## 决策理由
1. 功能最全面，满足复杂需求
2. 与K8s深度集成
3. 强大的流量管理能力
4. 企业级支持(Google/IBM)

## 实施计划
1. 阶段1: 试点部署(2周)
2. 阶段2: 核心服务迁移(4周)
3. 阶段3: 全量接入(8周)
```

### ADR: 数据库分片策略

```markdown
# ADR-XXX: 订单数据库分片策略

## 状态
已接受

## 背景
订单表数据量达到10亿+，单机PostgreSQL性能瓶颈:
- 查询延迟增加
- 写入吞吐量不足
- 备份时间过长

## 决策
采用 **按用户ID哈希分片** 策略。

## 选项分析

### 选项1: 按时间分片
- 优点: 热数据隔离，归档方便
- 缺点: 最新分片写入压力大，跨时间查询复杂

### 选项2: 按用户ID哈希分片
- 优点: 写入均匀分布，用户数据局部性好
- 缺点: 跨用户查询需要聚合

### 选项3: 按地理位置分片
- 优点: 合规友好，就近访问
- 缺点: 用户迁移复杂，分布不均

## 决策理由
1. 用户查询占90%，局部性重要
2. 写入负载均匀分布
3. 实现相对简单

## 分片方案
- 分片键: user_id
- 分片数量: 16
- 分片算法: user_id % 16
- 路由层: 应用层路由 + 中间件(Vitess)

## 后果
- 跨分片查询需要应用层聚合
- 需要维护分片映射表
- 扩容需要重新分片
```

## ADR管理最佳实践

### 1. 编号规则
```
ADR-001: 第一个决策
ADR-002: 第二个决策
ADR-003: ...
```

### 2. 目录结构
```
docs/
└── adr/
    ├── README.md           # ADR索引
    ├── ADR-001-database.md
    ├── ADR-002-api-gateway.md
    ├── ADR-003-authentication.md
    └── template.md         # 模板文件
```

### 3. 索引文件 (README.md)
```markdown
# 架构决策记录索引

## 活跃决策

| 编号 | 标题 | 状态 | 日期 |
|------|------|------|------|
| [ADR-001](ADR-001-database.md) | 数据库选型 | 已接受 | 2024-01-15 |
| [ADR-002](ADR-002-api-gateway.md) | API网关选型 | 已接受 | 2024-02-01 |

## 已废弃决策

| 编号 | 标题 | 废弃原因 | 替代方案 |
|------|------|----------|----------|
| ADR-003 | 使用Redis 5 | 版本过旧 | ADR-010 |

## 按主题分类

### 数据架构
- ADR-001: 数据库选型
- ADR-005: 分片策略
- ADR-008: 缓存策略

### 服务架构
- ADR-002: API网关
- ADR-004: 服务网格
- ADR-006: 消息队列

### 安全架构
- ADR-003: 认证方案
- ADR-007: 加密策略
```

### 4. 状态流转
```
提议中 → 已接受 → (已废弃 或 已替代)
           ↓
        执行中
```

### 5. 定期回顾
- 每季度回顾所有活跃ADR
- 评估决策是否仍然有效
- 更新过时决策状态

## ADR工具推荐

### adr-tools (命令行)
```bash
# 安装
brew install adr-tools

# 创建新ADR
adr new "使用PostgreSQL作为主数据库"

# 更新状态
adr accept ADR-001

# 生成索引
adr generate index > docs/adr/README.md
```

### ADR Log (在线工具)
- https://adr.github.io/
- 可视化ADR管理

### 集成到代码仓库
```yaml
# .github/workflows/adr-check.yml
name: ADR Check
on: [pull_request]
jobs:
  check:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v2
      - name: Check ADR format
        run: |
          for f in docs/adr/ADR-*.md; do
            if ! grep -q "## 状态" "$f"; then
              echo "Missing status in $f"
              exit 1
            fi
          done
```
