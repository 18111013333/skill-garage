# 代码审查清单

## 一、代码质量检查

### 代码风格
- [ ] 命名规范：变量、函数、类命名清晰有意义
- [ ] 代码格式：缩进、空格、换行符合规范
- [ ] 注释质量：关键逻辑有注释，注释准确有用
- [ ] 代码长度：单文件不超过500行，单函数不超过50行
- [ ] 无重复代码：DRY原则，重复代码抽取为公共函数

### 代码逻辑
- [ ] 逻辑正确：代码实现符合需求
- [ ] 边界处理：边界条件、空值、异常情况处理
- [ ] 循环正确：循环条件、终止条件、无死循环
- [ ] 条件完整：if-else覆盖所有情况
- [ ] 无冗余逻辑：删除调试代码、注释掉的代码

### 性能考虑
- [ ] 算法效率：时间复杂度可接受
- [ ] 内存使用：无内存泄漏，大对象及时释放
- [ ] 数据库：查询优化，避免N+1问题
- [ ] 缓存：合理使用缓存
- [ ] 并发：线程安全，无竞态条件

## 二、安全检查

### 输入验证
- [ ] 参数校验：所有外部输入都经过验证
- [ ] 类型检查：类型转换安全
- [ ] 长度限制：字符串、数组长度限制
- [ ] 范围检查：数值范围验证
- [ ] 特殊字符：SQL注入、XSS防护

### 权限控制
- [ ] 认证检查：需要认证的接口有验证
- [ ] 授权检查：权限验证正确
- [ ] 敏感操作：敏感操作有二次确认
- [ ] 数据隔离：用户只能访问自己的数据

### 数据安全
- [ ] 敏感数据：密码、密钥加密存储
- [ ] 日志脱敏：日志不包含敏感信息
- [ ] 传输加密：HTTPS传输
- [ ] SQL安全：使用参数化查询

## 三、可维护性检查

### 代码结构
- [ ] 模块化：职责单一，高内聚低耦合
- [ ] 分层清晰：表现层、业务层、数据层分离
- [ ] 依赖管理：依赖注入，避免硬编码依赖
- [ ] 接口设计：接口清晰，参数合理

### 错误处理
- [ ] 异常捕获：异常被正确捕获和处理
- [ ] 错误信息：错误信息友好，不暴露敏感信息
- [ ] 日志记录：关键操作有日志
- [ ] 降级处理：有合理的降级方案

### 可测试性
- [ ] 单元测试：核心逻辑有单元测试
- [ ] 测试覆盖：测试覆盖率达标
- [ ] Mock友好：依赖可Mock
- [ ] 边界测试：边界情况有测试

## 四、文档检查

### 代码文档
- [ ] 函数注释：公共函数有注释说明
- [ ] 参数说明：参数含义、类型、范围说明
- [ ] 返回值说明：返回值含义说明
- [ ] 示例代码：复杂函数有使用示例

### API文档
- [ ] 接口说明：接口功能说明
- [ ] 参数文档：请求参数说明
- [ ] 响应文档：响应格式说明
- [ ] 错误码：错误码列表

### 变更文档
- [ ] CHANGELOG：变更记录
- [ ] 升级指南：不兼容变更说明
- [ ] 配置说明：新增配置项说明

## 五、审查流程

### 审查前准备
1. 理解需求背景
2. 阅读相关文档
3. 了解技术方案

### 审查过程
1. 先整体后细节
2. 关注核心逻辑
3. 标记问题和建议
4. 区分必须修改和建议修改

### 审查反馈
```
## 代码审查反馈

### 必须修改 🔴
- [ ] 问题1：描述 + 原因 + 建议
- [ ] 问题2：描述 + 原因 + 建议

### 建议修改 🟡
- [ ] 建议1：描述 + 原因 + 建议
- [ ] 建议2：描述 + 原因 + 建议

### 优点 👍
- 优点1
- 优点2

### 问题讨论 💬
- 问题1：描述
```
